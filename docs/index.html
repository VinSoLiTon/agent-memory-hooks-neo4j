Agent Memory Hooks

A two-stage memory layer for Claude Code, Codex, Cursor, and Gemini CLI — backed by Neo4j.

claude_code codex cursor gemini Neo4j Python 3.10+

Overview

Four agent CLIs write into one shared Neo4j graph. The system has two stages that run on completely different cadences:

ONLINE  Stage 1 — Hooks

Fire on every session event from any CLI. They capture what happened — prompts, tool calls, tool responses — as a per-session linked list of :Event nodes, with secrets scrubbed and opted-out directories dropped. They also inject memory back into the session at SessionStart and on each user prompt.

Latency-critical: runs inline with the agent. Touches only the two MERGE-supporting UNIQUE constraints; never does schema-heavy work.

OFFLINE  Stage 2 — Dream phase

Runs on a schedule (Windows Task Scheduler, daily 3 PM). Reads sessions newer than their last_dreamed_at watermark, feeds the trimmed event log to an LLM (Anthropic / OpenAI / local Ollama), and decides what's worth remembering — distilling durable markdown :Memory nodes that future sessions automatically receive.

Not latency-critical. Can spend real tokens and seconds per session.

The split is the core idea: hooks record everything cheaply; the dream phase curates slowly. Recall combines fulltext and vector search and boosts in-project hits. Every node carries a client property so the four CLIs share one graph without colliding.

System block diagram

Solid grey edges = online capture/recall path. Dashed purple edges = offline dream path. Everything funnels through one Neo4j instance; the CLI, dashboard, and pattern detector are read/curate surfaces over the same graph.

Graph schema

Each session is a linked list of events. The dream phase attaches memories with full provenance.

(:Session {session_key, session_id, client, created_at, last_dreamed_at})
  -[:FIRST_EVENT]->  (:Event)            // head of the chain
  -[:LATEST_EVENT]-> (:Event)            // tail (watermark anchor)
  -[:DREAMED]->      (:Memory)           // provenance: this session produced it

(:Event {event_id, event_name, client, timestamp, tool_name, tool_input,
         tool_use_id, tool_response, prompt, model, source, turn_id,
         last_assistant_message, stop_hook_active, transcript_path,
         transcript, cwd})
  -[:NEXT]-> (:Event)                    // chronological successor

(:Memory {path, content, updated_at, project, archived, archived_at,
          access_count, last_accessed_at, embedding, embedding_model,
          embedding_dim, consolidated_from, promoted_from_pattern})
  -[:DERIVED_FROM]-> (:Session)

constraints: Session.session_key UNIQUE, Event.event_id UNIQUE, Memory.path UNIQUE
indexes:     memory_fulltext (Memory.content, Memory.path),
             memory_embeddings (vector on Memory.embedding, cosine),
             memory_project (Memory.project), session_id_lookup (Session.session_id)
KeyWhy it exists
session_key = "<client>:<session_id>" Composite so two CLIs can reuse the same raw session_id without merging into one chain. The original Session.session_id UNIQUE constraint is dropped by the migration.
event_id = "<client>_<session_id>_<ts>_<event>" Client-namespaced and timestamped so events never collide; MERGE-safe.
last_dreamed_at Watermark — the dream phase only re-reads events newer than this, and always advances it (even when 0 memories result) so low-signal sessions aren't re-dreamed forever.
project on :Memory Set for project/ & general/ memories; always null for cross-project profile/ & tools/. Drives the recall boost.

Implementation — Stage 1 · Capture

hooks/log_event.py

One shared writer for all four clients; --client tags the origin. The CLIs' hook payloads share almost all fields (session_id, hook_event_name, tool_name, tool_input, tool_response, prompt, …); Codex adds turn_id, tool_use_id, last_assistant_message, stop_hook_active.

Append algorithm

Each event is appended to the session's linked list in one idempotent Cypher write that relinks the LATEST_EVENT pointer:

MERGE (s:Session {session_key: $session_key})
  ON CREATE SET s.created_at = $timestamp, s.session_id = $session_id, s.client = $client
WITH s
CREATE (e:Event $event_props)
WITH s, e
OPTIONAL MATCH (s)-[old_latest:LATEST_EVENT]->(prev:Event)
DELETE old_latest
WITH s, e, prev
FOREACH (_ IN CASE WHEN prev IS NOT NULL THEN [1] ELSE [] END |
    CREATE (prev)-[:NEXT]->(e))          // link onto the tail
FOREACH (_ IN CASE WHEN prev IS NULL THEN [1] ELSE [] END |
    CREATE (s)-[:FIRST_EVENT]->(e))      // first event of the session
CREATE (s)-[:LATEST_EVENT]->(e)          // new tail

Design points

  • Minimal hot-path schema. The hook only runs CREATE CONSTRAINT IF NOT EXISTS for the two UNIQUE constraints its MERGEs depend on (session_key, event_id). Everything heavier — dropping legacy constraints, indexes, backfills — lives in njhook migrate.
  • Field hygiene. tool_response is capped at 4000 chars; tool_input is JSON-stringified then scrubbed. Empty dicts/strings are kept (present-but-empty ≠ missing); only None fields are stripped.
  • Transcripts off by default. Set HOOKS_CAPTURE_TRANSCRIPT=1 to opt in (capped at HOOKS_TRANSCRIPT_MAX_CHARS). They're duplicate data — every event is already stored.
  • Never blocks the agent. The whole hook is wrapped so any exception prints to stderr and returns; a Neo4j outage degrades to "no capture", not a broken CLI.

Privacy filters

hooks/privacy.py

Runs on every captured event, in two stages — both best-effort (an exception never blocks capture):

  1. CWD opt-out. Drop the event entirely if cwd (or any parent) matches the blocklist from HOOKS_OPT_OUT_PATHS (semicolon-separated) or ~/.njhook/optout.txt. Match is a case-folded, forward-slashed prefix compare so it's stable across Windows/POSIX.
  2. Secret scrubbing. Regex-redact secrets in every captured string before write.
Two pattern tiers The patterns are split so the same regexes aren't over-applied where they'd cause false positives:
  • High-confidence — distinctive real-secret shapes: sk-ant-…, generic sk-…, GitHub gh[oprsu]_…, AWS AKIA… / aws_secret_access_key, Slack xox[baprs]-…, Stripe, JWT eyJ…, PEM private-key blocks.
  • Heuristic — Bearer <token> and KEY=VALUE for sensitive-looking names. Useful for scrubbing captured events but too eager elsewhere.
scrub() (capture path) applies both. scrub_high_confidence() applies only the first tier and is used by the dream quality gate — so a memory documenting HOOKS_NEO4J_PASSWORD=password is accepted, while a real sk-ant- key is still rejected.

Memory injection (recall)

hooks/inject_memory.py

The other half of Stage 1: handing relevant memories back to the agent. Two trigger points, both emitting {"hookSpecificOutput": {"hookEventName": …, "additionalContext": …}}:

SessionStart

  • Loads three buckets ordered by recency: profile/* (≤ INJECT_PROFILE_LIMIT), tools/* (≤ INJECT_TOOLS_LIMIT), and memories tagged with the current project (≤ INJECT_PROJECT_LIMIT).
  • Concatenated under a markdown header until INJECT_CHAR_BUDGET (default 4000) is hit, then a _(further memories omitted…)_ marker is appended. Archived memories are excluded.

UserPromptSubmit / BeforeAgent / beforeSubmitPrompt

Hybrid recall against the prompt:

  1. Fulltext — Lucene over (m.content, m.path). Reserved characters are escaped so a prompt like "what does -x mean?" can't raise a parser error; on any failure it returns [] so the vector path still runs. If the raw query returns nothing, it retries with the prompt's content words OR-joined (stopwords stripped).
  2. Vector — ANN over the memory_embeddings index against the verbatim prompt. Returns [] when EMBED_PROVIDER is unset or the index isn't populated, so recall degrades gracefully to fulltext-only.
  3. Fusion — Reciprocal Rank Fusion (k=60) merges the two ranked lists; an in-project match adds a small INJECT_PROJECT_BOOST nudge in the same magnitude as RRF scores.

Returned memories get a best-effort access_count + last_accessed_at bump (used by archive and stats); failure there never blocks recall.

Project derivation

hooks/project.py

Single source of truth for "what project is this cwd?", shared by injection (scoping/boost) and dream (tagging). Walk up from cwd to the nearest .git ancestor and use that folder's lower-cased name; fall back to the leaf folder name. dominant_project() picks the most frequent project across a session's event cwds.

Embeddings

hooks/embeddings.py

Optional semantic layer; controlled by EMBED_PROVIDER (openai | ollama | unset). When unset, the whole system runs fulltext-only.

ProviderDefault modelDimNotes
openaitext-embedding-3-small1536via openai SDK
ollamanomic-embed-text:latest768local /api/embed, batched, no key
  • Batched — all texts in one request. memory_text() embeds path + "\n\n" + content so file-naming signal contributes to similarity.
  • Input cap — every text is truncated to EMBED_MAX_CHARS (6000) first. Small-context backends (nomic ~2048 tokens) return HTTP 400 on overlong input, and since the call is batched, one oversize memory would have silently dropped vectors for the whole batch.
  • Dim tracking — embedding_model + embedding_dim are stored on each memory so njhook reindex can detect a model swap and rebuild the vector index.

Implementation — Stage 2 · Dream phase

dream/dream.py

Reads sessions with events past their watermark, distills memories via the chosen provider, upserts them.

Fetch — watermark + explicit chain walk

  1. List candidate sessions with their last_dreamed_at and the timestamp of their LATEST_EVENT — one cheap hop, no chain walk.
  2. Skip any session whose latest event is ≤ watermark (nothing new) or predates --since.
  3. For the rest, walk the NEXT chain one hop at a time, then filter to events past the watermark.
Why single-hop, not a variable-length path The obvious query MATCH (s)-[:FIRST_EVENT|NEXT*0..]->(e) materializes a path for every reachable event and blows Neo4j's transaction-memory pool (MemoryPoolOutOfMemoryError, ~2.7 GiB default) on any session past ~150 events. Walking the linked list explicitly is a series of O(1) lookups with bounded memory; a seen set guards corrupted/branching chains from looping forever. A 1741-event session fetches in ~3 s.

Render — trim to signal

Prompts are kept in full (highest signal); tool input collapses to its command/file_path field, and tool responses collapse to an 80-char one-liner. Smaller local models drown in raw transcript dumps.

Write — quality gate, upsert, embed, advance watermark

  • Memories pass through the quality gate; rejects are logged, the run continues.
  • profile/ & tools/ paths are forced project = null (cross-project); everything else is tagged with the session's dominant project.
  • If embeddings are enabled, all bodies are embedded in one batch and the vector index is lazily created at the model's dimension.
  • Upsert is MERGE (m:Memory {path}) + MERGE (s)-[:DREAMED]->(m) + MERGE (m)-[:DERIVED_FROM]->(s).
  • The watermark advances to the latest dreamed event's timestamp even if zero memories were produced.
Token ceiling DREAM_MAX_TOKENS defaults to 16384 (was 4096). 4096 truncated the LLM's memory JSON mid-string on large sessions, producing unparseable output. It's a ceiling — you only pay for tokens generated.

Dream providers

dream/providers.py

Three adapters, one shape: dream(transcript, existing, system, model, max_tokens) -> list[dict]. Precedence: --provider flag > $DREAM_PROVIDER > anthropic.

All provider-specific JSON recovery lives here so dream.py stays clean. Imports are lazy so an unselected provider's SDK need not be installed.

Per-provider prompts

dream/prompts.py

Shared core rules (what counts as a memory, path conventions, a hard project-discrimination rule, merge/dedup rules) compose into two system prompts:

  • Frontier (Anthropic/OpenAI) — lean, no examples; these models follow abstract rules well.
  • Ollama — same rules plus a concrete few-shot example anchoring format and the cross-project vs scoped distinction, since 14B-class models need stricter cues.

DREAM_JSON_SCHEMA pins the output shape ({"memories":[{path, content}]}) with the path regex ^(profile|tools|project|general)/…\.md$ — passed straight to Ollama's format=.

Quality gates

dream/quality.py

Every memory is validated before any DB write, so a malformed model response can't corrupt the graph:

  • Path matches ^(profile|tools|project|general)/.+\.md$.
  • Body starts with --- YAML frontmatter containing title: and a valid kind: (one of profile/tool/project/general).
  • Body length in [DREAM_MEMORY_MIN_CHARS, DREAM_MEMORY_MAX_CHARS] (30–20000).
  • Secret defense-in-depth — reject if scrub_high_confidence(body) != body (the model can regenerate a real key from scrubbed context). Uses the high-confidence tier only, so config docs survive.

validate_batch() filters to the valid memories and logs each rejection to stderr.

Consolidate & archive

dream/consolidate.py

consolidate()

Greedy de-duplication: query the vector index for memory pairs above a cosine threshold (default 0.92), ask the LLM to merge the strongest pair, write the merged memory and rewire provenance (every :Session that DREAMED either original now DREAMs the merged one), then DETACH DELETE the originals. Repeats up to --rounds. consolidated_from records the source paths. Requires both an embedding provider and a dream provider.

archive()

Set archived=true on memories whose last_accessed_at AND updated_at are both older than --stale-days (default 60). profile/* is exempt. Recall already filters archived=false, so archived memories vanish from sessions but remain queryable via list --include-archived / unarchive. Needs no LLM.

Pattern detection

detect/patterns.py

Three detectors surface repeated signal across captured sessions; each result carries a stable 6-char id (sha1 of defining content) so you can reference it across runs.

DetectorWhat it finds
repeated_commands()Exact-normalized Bash/shell commands run ≥ min_count times (flags kept — pytest -x ≠ pytest -xvs).
hot_files()File paths Read/Edit/Write'd ≥ min_count times, with a per-tool breakdown.
prompt_clusters()User prompts greedily clustered by embedding cosine similarity (≥ threshold, default 0.8). Requires EMBED_PROVIDER.

draft_memory_from_pattern() deterministically (no LLM) converts a pattern into a draft :Memory — a command → tools/<binary>/usage.md, a hot file → project/hot-file-*.md, a prompt cluster → general/recurring-*.md, each tagged promoted_from_pattern. Surface and promote via njhook patterns --promote <id>.

Schema migration

hooks/schema.py

Run from njhook migrate (never the hot path). Three separate transactions — Neo4j forbids mixing schema and data writes:

  1. Drop legacy constraints — finds and drops the old Session.session_id UNIQUE via SHOW CONSTRAINTS.
  2. Create constraints + indexes — session_key/event_id/path UNIQUE, the fulltext index, memory_project, and session_id_lookup.
  3. Backfill — set session_key = "<client>:<session_id>" on pre-existing sessions.

Idempotent — re-running after a pull is safe and quick.

Usage — Install & setup

# 1. dependencies
pip install -r requirements.txt
pip install -r dream/requirements.txt        # dream phase
pip install -r dashboard/requirements.txt    # optional web UI

# 2. point at Neo4j (defaults shown)
export HOOKS_NEO4J_URI=bolt://localhost:7687
export HOOKS_NEO4J_USER=neo4j
export HOOKS_NEO4J_PASSWORD=password

# 3. run the schema migration once (idempotent)
./njhook.cmd migrate

# 4. (optional) enable semantic recall + local dreaming
export EMBED_PROVIDER=ollama
ollama pull nomic-embed-text

# 5. verify the whole stack
./njhook.cmd health

On POSIX hosts, run python cli/njhook.py <subcommand> directly instead of the njhook.cmd launcher.

Per-client wiring

The hooks are wired at the project level — open this directory in any CLI and they fire automatically.

CLIConfig fileNotes
Claude Code.claude/settings.jsonJust run Claude Code in this dir.
Codex.codex/hooks.jsonSet [features] codex_hooks = true in ~/.codex/config.toml.
Cursor.cursor/hooks.json (or legacy settings.json)Open the folder.
Gemini CLI.gemini/settings.jsonUses BeforeAgent as the user-prompt analog.

Each config calls a per-client wrapper (.cmd on Windows / .sh on POSIX) that forwards the hook's stdin JSON to the shared Python with --client <name>. For global capture across all projects, merge a hooks block into your user-level config pointing at the absolute path of the wrappers in this repo.

Windows gotcha In Claude Code hook commands, bash strips unquoted Windows backslashes — quote the path and use forward slashes.

Running the dream phase

python dream/dream.py                          # default provider (anthropic)
python dream/dream.py --since 24h              # only events newer than 24h / 7d / 30m
python dream/dream.py --dry-run                # preview, don't write
python dream/dream.py --provider ollama        # local, no API key
python dream/dream.py --provider openai --model gpt-4o-mini
python dream/dream.py --session claude_code:<id>
python dream/dream.py --consolidate            # LLM-merge near-duplicates
python dream/dream.py --archive --stale-days 60

Scheduled (Windows Task Scheduler) — daily 3 PM

$action  = New-ScheduledTaskAction -Execute "C:\Projects\njhook\dream\run_dream.cmd"
$trigger = New-ScheduledTaskTrigger -Daily -At 3pm
$settings = New-ScheduledTaskSettingsSet -StartWhenAvailable `
  -ExecutionTimeLimit (New-TimeSpan -Minutes 30) -MultipleInstances IgnoreNew
Register-ScheduledTask -TaskName "njhook-dream-nightly" `
  -Action $action -Trigger $trigger -Settings $settings -RunLevel Limited

StartWhenAvailable=True means a run missed while the machine is off catches up on next power-on. Logs land in dream/logs/dream_YYYY-MM-DD.log; njhook health surfaces the last exit code and a dream-freshness check.

Web dashboard

dashboard/app.py

python dashboard/app.py                    # http://localhost:5000 (read-only)
DASHBOARD_WRITE=1 python dashboard/app.py  # enable edit / delete / archive

Flask UI to browse memories, walk session event chains, and run hybrid search. Mutations are off by default (a "read-only" pill shows in the header) — even bound to 127.0.0.1, any local process can hit it, so a casual click shouldn't mutate the graph. Routes: /memories, /memory/<path>, /sessions, /session/<sid>, /search, /stats.

Common workflows

# after install
./njhook.cmd migrate && ./njhook.cmd health

# after enabling EMBED_PROVIDER for the first time
./njhook.cmd embed-backfill
./njhook.cmd reindex                 # after switching embedding models

# daily inspection
./njhook.cmd stats

./njhook.cmd patterns --since 7d

# promote a recurring pattern to a memory
./njhook.cmd patterns --promote 135e52        # preview
./njhook.cmd patterns --promote 135e52 -y     # write

# clean up
./njhook.cmd consolidate --dry-run            # preview merges
./njhook.cmd consolidate
./njhook.cmd archive --stale-days 60

# disaster prep / recovery
./njhook.cmd backup --with-embeddings --out backup.json
./njhook.cmd restore --in backup.json --with-embeddings --dry-run
./njhook.cmd restore --in backup.json --with-embeddings
